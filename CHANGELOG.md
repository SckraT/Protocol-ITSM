# Changelog

Все значимые изменения проекта фиксируются в этом файле.
Формат основан на [Keep a Changelog](https://keepachangelog.com/ru/1.1.0/).

## [1.6.0] — 2026-05-31

### Добавлено
- **Счётчик просроченных задач в заголовке вкладки** — если есть незакрытые задачи с истёкшим
  сроком, заголовок браузерной вкладки меняется на `(N) Протокол совещаний`. Обновляется при
  каждой перерисовке таблицы и загрузке данных.
- **Inline-редактирование темы задачи** — двойной клик по ячейке темы открывает поле ввода
  прямо в строке таблицы. `Enter` сохраняет (PUT `/api/items/{id}`), `Escape` отменяет.
  Оптимистичный UI: изменение отражается мгновенно, откат — при ошибке сети.

### Исправлено / улучшено
- BACKLOG.md: отмечены выполненные пп. 12 (тёмная тема) и 13 (bulk-операции); добавлены
  пп. 14–15 (горячие клавиши, экспорт в Excel) и раздел стратегических задач (пп. 16–19).

## [1.5.0] — 2026-05-31

### Изменено (рефакторинг фронтенда)
- **Vite + TypeScript**: фронтенд перенесён в `frontend/` и собирается через Vite 5 + TypeScript 5.
  Результат сборки (`static/`) не изменился с точки зрения работы приложения.
- **Разбивка JS на модули** (`src/`):
  - `types.ts` — интерфейсы `Item`, `Executor`, `Department`, `Status` и вспомогательные типы.
  - `utils.ts` — чистые функции (`fmtDate`, `esc`, `plural`, `dueInfo`) и константы (`STATE_LABEL`, `PRIORITY_CONFIG`…).
  - `api.ts` — все HTTP-вызовы к бэкенду с типизированными возвратами.
  - `state.ts` — единый объект `AppState`, `saveViewState`, `loadViewState`.
  - `theme.ts` — управление тёмной/светлой темой.
  - `toast.ts` — уведомления.
  - `filters.ts` — `applyFilters`, `applySort`, DOM-события фильтров.
  - `render.ts` — `render`, `renderRow`, `renderAccordion`, вспомогательные DOM-функции.
  - `loader.ts` — `loadAll` (оркестрация загрузки данных).
  - `modals.ts` — модалки задачи и истории, быстрый статус, смена состояния, удаление.
  - `bulk.ts` — bulk-операции (выбор строк, массовые изменения).
  - `refs.ts` — вкладка «Справочники» (CRUD отделов и исполнителей), `loadRefs`.
  - `confirm.ts` — диалог подтверждения.
  - `main.ts` — точка входа: инициализация, обработчики событий, экспозиция в `window`.
- **Разбивка CSS на 5 файлов** (`src/styles/`):
  `theme.css`, `base.css`, `components.css`, `table.css`, `print.css`.
- **Dockerfile**: multi-stage сборка — Node-стейдж собирает фронтенд, Python-стейдж запускает бэкенд.

## [1.4.0] — 2026-05-31

### Добавлено
- **Пагинация**: таблица задач разбита на страницы. Панель пагинации появляется автоматически, когда задач больше одной страницы, и скрывается при малом числе записей.
  - Выбор размера страницы: 10 / 20 / 50 / Все (сохраняется в localStorage).
  - Умные кнопки навигации: `‹ 1 … 4 5 6 … 12 ›` с многоточием при большом числе страниц.
  - Счётчик `1–20 из 43`.
  - Нумерация строк глобальная (не сбрасывается на каждой странице).
  - При смене фильтра, поиска или сортировки — сброс на страницу 1.

  - Панель скрыта при печати.
  - Адаптивный вид на мобильных.

- **Улучшенный калькулятор просрочки**:
  - «Завтра» вместо «через 1 день».
  - Для просроченных задач (≥ 3 дней) показывает рабочие дни в скобках: «просрочено на 6 дней (5 рабочих)» — исключает субботу и воскресенье.
  - Относительное время для будущих дат: «через 2 нед.» (8–30 дней), «через 2 мес.» (> 30 дней).
  - Закрытые задачи по-прежнему не показывают срок.

## [1.0.0] — 2026-05-30

### Добавлено
- **Оптимистичный UI**: основные операции (удаление задачи, смена состояния, добавление быстрого статуса) больше не вызывают полную перезагрузку `loadAll()`. Изменения применяются мгновенно в UI, затем синхронизируются с API.
- **Сохранение состояния DOM**: позиция прокрутки таблицы и раскрытые аккордеоны теперь сохраняются при операциях и восстанавливаются после обновления.
- **Откат при ошибках API**: если запрос к API упал, оптимистичное изменение откатывается и показывается сообщение об ошибке.

### Изменено
- Удаление задачи (функция `deleteItem`): вместо вызова `loadAll()` удаляет строку из таблицы, обновляет счётчики дашборда, отправляет DELETE запрос.
- Смена состояния (функция `changeState`): обновляет объект в памяти и счётчики без перерисовки всей таблицы.
- Добавление быстрого статуса (функция `saveQs`): добавляет статус в `recent_statuses` в памяти, перерисовывает только нужную строку.
- Удаление статуса из истории (функция `deleteStatus`): больше не вызывает `loadAll()`, только обновляет историю в модальном окне.

### Исправлено
- Потеря фокуса после любой операции с задачей (теперь состояние DOM восстанавливается)
- Потеря позиции прокрутки при добавлении/редактировании/удалении задач

- **Статусы на главном экране** теперь выводятся компактно: одна строка с последним статусом (дата + текст в одну строку с обрезкой по ширине, полный текст — в тултипе) и бейдж «+N» с числом скрытых записей. Это выравнивает высоту строк таблицы. Клик по «+N» (как и по ссылке «вся история») открывает полную историю.

### Добавлено
- API `GET /api/items`: поле `status_count` — общее число статусов задачи (используется счётчиком скрытых записей в аккордеоне). Доп. запрос к БД выполняется только когда статусов больше трёх.

## [0.6.0] — 2026-05-30

### Добавлено
- **Сводка-дашборд** над таблицей: три плитки со счётчиками — «В работе», «Просрочено», «Закрыто за неделю». Обновляются при каждой загрузке данных.
  - «Просрочено» — задачи не в состоянии «Закрыто», у которых срок истёк. Число красное при ненуле, серое при нуле.
  - «Закрыто за неделю» — закрытые задачи, у которых дата последнего статуса попадает в последние 7 дней (прокси для `closed_at`, которого нет в схеме).

## [0.5.1] — 2026-05-30

### Добавлено
- **Редактирование записей справочников.** У каждого отдела и исполнителя появилась кнопка ✏️ — открывает инлайн-редактирование прямо в строке: для отдела меняется название, для исполнителя — имя и отдел (выпадающий список). Сохранение по кнопке или `Enter`, отмена по `×` или `Escape`.
- API: `PUT /api/departments/{id}` и `PUT /api/executors/{id}`. Оба возвращают `409` при попытке занять имя, уже использованное другой записью; `404` — если запись не найдена.

## [0.5.0] — 2026-05-30

### Добавлено
- **Тосты** (всплывающие уведомления) вместо браузерных `alert()` — для успеха и ошибок операций (сохранение, удаление, импорт, дубли в справочниках).
- **Кастомный диалог подтверждения** вместо браузерного `confirm()` при удалении задач, статусов, отделов и исполнителей — в стиле приложения.
- **Обработка ошибок и состояний загрузки**: при загрузке таблицы показывается спиннер; при сбое сети — сообщение с кнопкой «Повторить». Все изменяющие запросы при ошибке показывают тост вместо тихого сбоя.
- **Относительная подпись срока**: «просрочено на N дней», «сегодня», «через N дней» (с правильным склонением). Срок «скоро истекает» (≤ 3 дней) подсвечивается жёлтым.
- **Закрытие модальных окон по `Escape`** (подтверждение → история → форма задачи).

### Изменено
- **Маркер приоритета** теперь содержит букву (В / С / Н) в дополнение к цвету — различим для дальтоников; добавлен `aria-label`.
- **Доступность**: кликабельные бейдж состояния и ссылка истории получили `role="button"`, фокус с клавиатуры и активацию по `Enter` / `Space`; добавлен видимый стиль фокуса.

## [0.4.0] — 2026-05-29

### Добавлено
- **Справочник отделов** — вкладка «Справочники» теперь содержит две секции: Отделы и Исполнители. Отделы добавляются/удаляются там же. При удалении отдела исполнители сохраняются, но теряют привязку.
- Поле **«Отдел»** у исполнителя — при добавлении в справочник можно выбрать отдел из списка.
- В форме задачи поле исполнителя заменено на **выпадающий список**, сгруппированный по отделам (`<optgroup>`). Записи без отдела — в группе «Без отдела».
- В таблице исполнители отображаются чипами с указанием отдела.
- Исполнители задачи хранятся по ID (а не по имени), миграция выполняется автоматически.
- Таблица `departments` в БД; колонка `department_id` в таблице `executors`.

## [0.3.0] — 2026-05-29

### Добавлено
- Вкладка **«Исполнители»** — справочник исполнителей с добавлением и удалением. Добавление по Enter или кнопке.
- Динамические поля исполнителей в форме задачи: по умолчанию одна строка, кнопка «+ добавить исполнителя» добавляет ещё; кнопка «−» убирает (последнюю строку только очищает).
- Автодополнение в полях исполнителей из справочника (HTML5 datalist).
- Удаление исполнителя из справочника не затрагивает уже привязанные задачи.

### Изменено
- Исполнители задачи хранятся как JSON-массив в колонке `executors_json` вместо двух отдельных полей `executor`/`executor2`.
- API: поле `executors` теперь массив строк; в CSV исполнители разделяются символом `|`.
- Автоматическая миграция БД: данные из `executor` и `executor2` переносятся в `executors_json`.

## [0.2.0] — 2026-05-29

### Добавлено
- **Поиск** по теме, тикету и исполнителям — строка в тулбаре, фильтрует в реальном времени.
- **Сортировка** по столбцам: кликом по заголовку (тикет, тема, состояние, срок, исполнитель). Повторный клик меняет направление.
- **Приоритет** задачи (Высокий / Средний / Низкий) — поле в форме, цветная точка в таблице, сортировка по приоритету.
- **Inline-смена состояния** — клик по бейджу «В работе / Отложено / Закрыто» циклически переключает состояние без открытия модалки.
- **Быстрый статус** — кнопка `+` в строке задачи раскрывает инлайн-форму с полями даты и описания. Enter сохраняет, Escape закрывает.
- **Экспорт в CSV** — кнопка «⬇ Экспорт» скачивает все задачи (с последним статусом), UTF-8 BOM для корректного открытия в Excel.
- **Импорт из CSV** — кнопка «⬆ Импорт» загружает задачи из файла того же формата.
- Зависимость `python-multipart` для обработки загрузки файлов.
- Автоматическая миграция БД: добавление колонки `priority`.

## [0.1.2] — 2026-05-29

### Добавлено
- **История статусов** для каждой задачи: статусы вынесены в отдельную таблицу
  `statuses` (связь один-ко-многим с `items`).
- Панель истории статусов с возможностью добавлять и удалять отдельные статусы.
- API-эндпоинты:
  - `GET /api/items/{id}/statuses` — история статусов задачи;
  - `POST /api/items/{id}/statuses` — добавить статус;
  - `DELETE /api/statuses/{id}` — удалить статус.

### Изменено
- При создании/редактировании задачи можно одновременно добавить новый статус.

### Миграции
- Автоматический перенос статусов из старой схемы (поля в таблице `items`)
  в новую таблицу `statuses` при первом запуске.
