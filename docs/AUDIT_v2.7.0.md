## 0. Резюме

Кодовая база **уже соответствует целевой архитектуре**, заявленной в задаче:
строго слоистый backend (`models → schemas → repositories → services → routers`),
async SQLAlchemy 2.0 + asyncpg, Pydantic v2, Alembic; frontend на Svelte 5 Runes
(`$state/$derived/$props`, сторы-классы в `.svelte.ts`), централизованный API-клиент,
персист фильтров в localStorage, Bits UI, Lucide, Tailwind.

«Грязного legacy для переписывания» нет. Полный rewrite дал бы высокий риск регрессий
(особенно в auth / meetings / audit / rate-limit — функционал, которого нет в README)
при близкой к нулю выгоде. Поэтому ниже — приоритизированный список **точечных** улучшений.

Большинство пунктов прошлого ТЗ-аудита (база v2.6.8; файл `Audit_Result_TZ.md` удалён как
исполненный) **уже закрыто** в релизе v2.6.9 (#20): CI/CD, тесты доменов, `iat` в JWT, типизация
роутеров и внутренних функций, предупреждение про localhost+CORS, LICENSE.

## 3. Приоритизированный список улучшений

### P1 — стоит сделать (реальные дефекты / расхождения со стандартом задачи)

1. **Дублирование и рассинхрон конфигурации Ruff.**
   Есть корневой [ruff.toml](../ruff.toml) с богатым набором правил (`E,F,W,I,N`,
   per-file-ignores) **и** `[tool.ruff]` в [backend/pyproject.toml](../backend/pyproject.toml)
   только с `line-length`/`target-version`.
   CI ([.github/workflows/ci.yml](../.github/workflows/ci.yml):38) запускает
   `ruff check app/` из каталога `backend/` → Ruff берёт **ближайший** конфиг
   (`pyproject.toml`) и **игнорирует корневой `ruff.toml`**. Значит правила `I` (isort),
   `N` (naming), `W` и per-file-ignores **в CI не применяются**.
   *Фикс:* перенести `[tool.ruff.lint]` (select/ignore/per-file-ignores) в
   `backend/pyproject.toml` ИЛИ убрать `[tool.ruff]` из pyproject и запускать ruff из корня.
   Один источник правды.

2. **Нет статического type-checker (mypy/pyright).**
   Задача требует строгой типизации и «избегать `Any`», но в тулинге/CI нет проверки типов —
   только Ruff (это линтер, не type-checker). Есть единичные `# type: ignore`
   (`item_service.py:146,183`) и утечки `any` через `Array.isArray`-narrowing в
   `client.ts:31`. *Фикс:* добавить `pyright` (или mypy) в `dev`-зависимости и отдельный
   шаг CI `pyright app/` (можно начать в режиме `|| true`, затем ужесточить).

3. **Нет фронтенд-тестов, хотя README обещает `npm run test`.**
   В [frontend/package.json](../frontend/package.json) нет скрипта `test` и нет Vitest —
   `npm run test` упадёт. Задача (п.6) и README прямо требуют frontend-тесты.
   *Фикс:* добавить Vitest + (опц.) `@testing-library/svelte`, покрыть как минимум
   чистую логику сторов (`items.filtered`, сортировка, `filters.persist`) и `client.ts`.

4. **Логирование через `print(...)`, а не через `logging`.**
   Задача и README требуют «структурированное логирование через Python logging».
   [main.py](../backend/app/main.py) и `_seed_first_admin` используют `print(..., flush=True)`.
   *Фикс:* завести `logging`-конфигуратор (уровни, формат), заменить `print` на `logger.*`.
   Поведение в Docker-логах не ухудшится, появится управляемость уровнями.

### P2 — желательно (надёжность / поддерживаемость; не дефекты)

5. **`datetime.utcnow()` устарел в Python 3.12.**
   [security.py](../backend/app/security.py):39 — `datetime.utcnow()` выдаёт
   DeprecationWarning и naive-datetime. *Фикс:* `datetime.now(timezone.utc)`
   (jose корректно сериализует aware-datetime в `exp`/`iat`).

6. **Дублирование логики фильтрации/сортировки BE ↔ FE.**
   Backend (`item_repository.list_with_filters`) умеет фильтровать по
   state/search/executor/dept/priority/meeting и пагинировать, но фронтенд грузит
   `page_size: 1000` и фильтрует/сортирует/пагинирует **в памяти** (`items.svelte.ts`,
   `ItemsTable.svelte`). Две независимые реализации одних правил → риск расхождения
   (напр. `ilike` на сервере vs `toLowerCase().includes` на клиенте). Для текущих
   объёмов это осознанный компромисс; зафиксировать его явно (комментарий/ADR) либо
   решить, какой слой — источник истины. **Менять без согласования не буду** (влияет на UX).

7. **Тесты на SQLite, прод на PostgreSQL.**
   [conftest.py](../backend/tests/conftest.py):29 — in-memory SQLite. Разница семантики
   (`ilike`/регистр, типы дат, каскады) не покрывается. Док. в README есть; как улучшение —
   опц. джоб CI с PostgreSQL-сервисом (`services: postgres`) для прогона тех же тестов.

8. **Миграции в `lifespan` через `subprocess`.**
   [main.py](../backend/app/main.py):57 запускает `alembic upgrade head` сабпроцессом при
   старте. Работает, но при многоворкерном деплое запускается в каждом воркере и связывает
   web-процесс с миграцией. *Опц.:* вынести в отдельный init-шаг (entrypoint/job),
   оставив в приложении только проверку версии.

9. **Refresh-токен без дедупликации параллельных 401.**
   `client.ts` `tryRefresh` не защищён от гонки: несколько одновременных запросов,
   получивших 401, инициируют несколько параллельных refresh. *Фикс:* кешировать
   «in-flight» промис refresh.

### P3 — косметика / на потом

10. **Оптимистичный merge в `items.update`** (`{ ...i, ...data }`) может временно класть в
    объект `Item` поля payload’а, которых нет в `Item` (напр. `executor_ids`). Безвредно
    (перетирается ответом сервера), но строже — мапить только известные поля.
11. **`subprocess`/`sys` импорты и инлайн-импорты внутри функций** (`main.py`,
    `csv_service`, conftest) — местами ради разрыва циклов; можно причесать, но не срочно.
12. **`docs/ARCHITECTURE.md` / `DEPLOY.md`** — проверить остаточные упоминания SQLite
    (пункт прошлого ТЗ, блок 5) — вне зоны кода, но влияет на онбординг.
13. **Bulk-операции — UX мелочи** (`BulkActions.svelte`). ✅ ИСПРАВЛЕНО (один
    агрегированный тост со счётчиком; `bulkChangeState`/`bulkRemove` с `silent`; покрыто
    тестами). Параллелизацию сознательно не делали — ломает оптимистичные snapshot/откат.
    Исходное описание ниже.
    Bulk реализован на клиенте
    циклом индивидуальных запросов (`PATCH`/`DELETE` на каждый id), серверного bulk-эндпоинта
    нет (и в контракте не заявлен — пробела нет). Наблюдения, **трогать без согласования
    не буду** (влияет на UX):
    - `applyState` всегда показывает `Обновлено: N` даже при частичном сбое (N — число
      попыток, не успехов); ошибки при этом всплывают отдельными тостами из `update`.
    - `deleteSelected` показывает N отдельных тостов «Задача удалена» (шум при удалении
      многих задач) — каждый `remove` тостит сам.
    - Запросы строго последовательны (`await` в цикле) — для больших выборок медленно.
    *Возможные фиксы (по согласованию):* агрегировать результат и тостить один раз
    (`Обновлено/удалено: успехов/всего`); опц. параллелить с учётом частичных ошибок.

### A. Безопасность

| ID | Приоритет | Находка | План |
|----|-----------|---------|------|
| **A1** | 🔴 ✅ | **Rate limiting за reverse-proxy не работал.** `middleware/rate_limit.py` брал `request.client.host` → за nginx это IP прокси, лимит `login (5/60)` считался суммарно по всем клиентам. **СДЕЛАНО (v2.7.0):** реальный IP из `X-Real-IP`/`X-Forwarded-For`; sliding window в **Redis** (ZSET), общий для воркеров и переживает рестарт; graceful fallback на in-memory без `REDIS_URL`/при сбое Redis.
