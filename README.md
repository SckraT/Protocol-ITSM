# Протокол совещания

Минималистичное self-hosted веб-приложение для ведения протокола совещаний:
учёт задач (тем), история статусов по каждой задаче, справочники отделов
и исполнителей. Замена ведению протокола в Excel.

## Возможности

### Задачи протокола
- Тема, номер тикета, приоритет (высокий / средний / низкий), состояние, срок, исполнители.
- **Inline-редактирование темы** — двойной клик по строке, `Enter` сохраняет, `Esc` отменяет.
- **Состояния**: в работе / отложено / закрыто. Переключаются через select прямо в строке таблицы.
- **История статусов** — у каждой задачи неограниченное число статусов (дата + описание).
  В таблице показываются 3 последних в аккордеоне, полная история — в отдельном окне.
- **Быстрый статус** прямо из строки таблицы (кнопка `+`).
- **Bulk-операции** — выделение нескольких задач и массовая смена состояния / удаление.

### Фильтрация и поиск
- **Дашборд** — кликабельные тайлы (Все / В работе / Отложено / Закрыто).
- **Расширенные фильтры** — по отделу, исполнителю и приоритету одновременно.
- **Поиск** по теме, тикету, исполнителю и отделу.
- **Сортировка** по любому столбцу.
- **Запоминание вида** — фильтры, поиск, сортировка сохраняются в `localStorage`.
- **Пагинация** — настраиваемый размер страницы.

### UX
- **Счётчик просроченных задач** в заголовке вкладки браузера — `(N) Протокол совещаний`.
- **Тёмная тема** — переключатель в шапке, выбор сохраняется.
- **Оптимистичный UI** — изменения применяются мгновенно, откат при ошибке сети.
- **Относительные сроки** — «просрочено на N дней», «сегодня», «через N дней».
- **Печать** — кнопка вызывает `@media print`-стиль с аккуратной вёрсткой.
- **Экспорт / импорт** в CSV (UTF-8 BOM, совместимо с Excel).

### Справочники
- CRUD отделов и исполнителей; исполнитель привязывается к отделу.
- Группировка по отделам, счётчики задач, поиск по имени.
- Инлайн-редактирование прямо в списке.

## Быстрый старт

Требуется установленный Docker.

```bash
docker compose up -d --build
```

Приложение доступно на **http://localhost:8000**.

### Полезные команды

```bash
docker compose logs -f        # логи в реальном времени
docker compose stop           # остановить
docker compose start          # запустить снова
docker compose down           # остановить и удалить контейнер (данные сохраняются)
```

**Данные** хранятся в `./data/protocol.db` (примонтированы через volume) —
пересборка контейнера их не теряет.

## Модель данных

Четыре таблицы SQLite. Схема создаётся и мигрируется автоматически при старте
(`init_db()` в [app.py](app.py)).

### `items` — задачи протокола

| Поле             | Тип     | Описание                                              |
|------------------|---------|-------------------------------------------------------|
| `id`             | INTEGER | PK                                                    |
| `topic`          | TEXT    | Тема задачи (обязательно)                             |
| `ticket`         | TEXT    | Номер тикета (необязательно)                          |
| `priority`       | TEXT    | `high` / `medium` / `low` / NULL                      |
| `state`          | TEXT    | `in_progress` / `postponed` / `closed`                |
| `due_date`       | TEXT    | Срок (ISO-дата `YYYY-MM-DD`)                          |
| `executors_json` | TEXT    | JSON-массив **ID** исполнителей, напр. `[3, 7]`       |
| `created_at`     | TEXT    | Дата создания                                         |

> Исполнители хранятся как массив ID, ссылающихся на `executors.id`.
> При отдаче через API разворачиваются в объекты `{id, name, department_name}`
> функцией `resolve_executors()`.

### `statuses` — история статусов

| Поле          | Тип     | Описание                                          |
|---------------|---------|---------------------------------------------------|
| `id`          | INTEGER | PK                                                |
| `item_id`     | INTEGER | FK → `items.id`, `ON DELETE CASCADE`              |
| `status_date` | TEXT    | Дата статуса                                      |
| `status_note` | TEXT    | Текст статуса                                     |
| `created_at`  | TEXT    | Время добавления                                  |

### `departments` — отделы

| Поле   | Тип     | Описание          |
|--------|---------|-------------------|
| `id`   | INTEGER | PK                |
| `name` | TEXT    | Название (UNIQUE) |

### `executors` — исполнители

| Поле            | Тип     | Описание                                            |
|-----------------|---------|-----------------------------------------------------|
| `id`            | INTEGER | PK                                                  |
| `name`          | TEXT    | ФИО / название (UNIQUE)                             |
| `department_id` | INTEGER | FK → `departments.id`, `ON DELETE SET NULL`         |

### Связи

```
departments 1───* executors
                      ▲
                      │ (по ID в executors_json)
items 1───* statuses  │
   └──────────────────┘  (многие-ко-многим через JSON-массив)
```

## REST API

Базовый префикс: `/api`. Все тела запросов/ответов — JSON (кроме импорта CSV).

Интерактивная документация: **http://localhost:8000/docs** (Swagger UI).

### Задачи

| Метод    | Путь                | Описание                                          |
|----------|---------------------|---------------------------------------------------|
| `GET`    | `/api/items`        | Список задач. Параметр `?state=` фильтрует.       |
| `POST`   | `/api/items`        | Создать задачу (опц. сразу первый статус).        |
| `PUT`    | `/api/items/{id}`   | Частичное обновление (только переданные поля).    |
| `DELETE` | `/api/items/{id}`   | Удалить задачу вместе со статусами.               |

Ответ `GET /api/items` для каждой задачи содержит:
- `executors` — массив объектов `{id, name, department_name}`;
- `recent_statuses` — до 3 последних статусов (от новых к старым);
- `status_count` — общее число статусов.

**Особенности `PUT`:** передаются только изменяемые поля; пустое значение очищает поле
(кроме `topic` и `state`, защищённых от обнуления); `executors` — массив ID.

### Статусы

| Метод    | Путь                          | Описание                          |
|----------|-------------------------------|-----------------------------------|
| `GET`    | `/api/items/{id}/statuses`    | Полная история статусов задачи.   |
| `POST`   | `/api/items/{id}/statuses`    | Добавить статус.                  |
| `DELETE` | `/api/statuses/{id}`          | Удалить отдельный статус.         |

### Отделы

| Метод    | Путь                       | Описание                                                  |
|----------|----------------------------|-----------------------------------------------------------|
| `GET`    | `/api/departments`         | Список отделов.                                           |
| `POST`   | `/api/departments`         | Создать отдел (409, если имя занято).                     |
| `DELETE` | `/api/departments/{id}`    | Удалить; у исполнителей отдела `department_id` → NULL.    |

### Исполнители

| Метод    | Путь                     | Описание                                             |
|----------|--------------------------|------------------------------------------------------|
| `GET`    | `/api/executors`         | Список с именем отдела (сортировка отдел→имя).       |
| `POST`   | `/api/executors`         | Создать (`{name, department_id?}`, 409 при дубле).   |
| `DELETE` | `/api/executors/{id}`    | Удалить из справочника (задачи не затрагиваются).    |

### Экспорт / импорт

| Метод  | Путь               | Описание                               |
|--------|--------------------|----------------------------------------|
| `GET`  | `/api/export/csv`  | Скачать все задачи в CSV (UTF-8 BOM).  |
| `POST` | `/api/import/csv`  | Загрузить CSV (`multipart/form-data`). |

## Устройство фронтенда

Фронтенд собирается с помощью **Vite 5 + TypeScript 5** из исходников в `frontend/src/`
в `static/` (результат сборки хранится в git — Docker использует его без пересборки при
отсутствии изменений).

### Поток данных

```
loadAll()
  ├── fetchItems() + fetchExecutors() + fetchDepartments()  [параллельно]
  ├── state.allItems / allExecutors / allDepartments
  ├── updateDashboardCounts()
  ├── updateTabTitle()   ← счётчик просроченных в заголовке вкладки
  └── render()
        ├── applyFilters(state.allItems)  ← currentFilter + searchQuery + adv filters
        ├── applySort()
        ├── renderPagination()
        └── renderRow() × N
```

### Глобальное состояние (`state.ts`)

| Поле              | Назначение                               |
|-------------------|------------------------------------------|
| `allItems`        | Кэш задач с сервера                      |
| `allExecutors`    | Справочник исполнителей                  |
| `allDepartments`  | Справочник отделов                       |
| `currentFilter`   | Фильтр по состоянию                      |
| `searchQuery`     | Строка поиска                            |
| `filterDept/Exec/Priority` | Расширенные фильтры              |
| `sortState`       | Колонка и направление сортировки         |
| `currentPage`     | Текущая страница                         |
| `pageSize`        | Размер страницы (сохраняется в localStorage) |
| `selectedItems`   | Выбранные для bulk-операций              |
| `openAccordions`  | Раскрытые аккордеоны (восстанавливаются после render) |

### Оптимистичный UI

Критичные действия (смена состояния, добавление статуса, inline-редактирование) применяются
к `state` немедленно, без ожидания сервера. При ошибке — откат и тост.

## Миграции БД

Все миграции выполняются в `init_db()` при старте и **идемпотентны**
(повторный запуск безопасен). Подход: проверка наличия колонки через
`PRAGMA table_info`, затем `ALTER TABLE`.

> При добавлении новой колонки — дописывайте миграцию в `init_db()`,
> не рассчитывая на пересоздание БД (у пользователя уже есть данные).

## Экспорт / импорт CSV

Формат столбцов (`CSV_FIELDS` в [app.py](app.py)):

```
id, topic, ticket, priority, state, due_date, executors, last_status_date, last_status_note
```

- **Кодировка**: UTF-8 с BOM — чтобы Excel корректно открывал кириллицу.
- **Исполнители**: в одной ячейке через ` | `, в формате `Отдел — Имя`.
- **Импорт**: `id` игнорируется (создаются новые записи); исполнители сопоставляются
  по имени, ненайденные создаются без отдела; импортируется только последний статус.

## Разработка

### Запуск без Docker

```bash
pip install -r requirements.txt
uvicorn app:app --reload --port 8000
```

БД создаётся в `data/protocol.db`. Путь переопределяется через `DB_PATH`.

### Разработка фронтенда (Vite dev server)

```bash
cd frontend
npm install
npm run dev        # dev-сервер на :5173, проксирует /api → localhost:8000
```

При этом бэкенд должен быть запущен отдельно (Docker или uvicorn).
После изменений собрать продакшн-билд:

```bash
npm run build      # → static/
```

После сборки нужна пересборка Docker-образа:

```bash
docker compose up -d --build
```

### Конвенции

- **Бэкенд**: каждый эндпоинт открывает соединение через `get_db()` и закрывает его
  перед возвратом. Комментарии — на русском.
- **Фронтенд**: TypeScript, строгий режим (`strict: true`). Новый код — в виде
  экспортируемых функций в подходящем модуле; инлайн `onclick` в HTML-строках
  только через `window` (экспозиция в `main.ts`).
- Любое заметное изменение фиксируется в [CHANGELOG.md](CHANGELOG.md)
  (формат [Keep a Changelog](https://keepachangelog.com/ru/1.1.0/)).

## Резервное копирование

Вся база — один файл `data/protocol.db`:

```bash
cp data/protocol.db backup-$(date +%Y%m%d).db
```

Восстановление — заменой файла при остановленном контейнере.
Альтернатива — выгрузка через `GET /api/export/csv`.
