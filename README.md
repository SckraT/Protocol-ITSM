# Протокол совещания

Минималистичное self-hosted веб-приложение для ведения протокола совещаний:
учёт задач (тем), история статусов по каждой задаче, справочники отделов
и исполнителей. Замена ведению протокола в Excel.

## Возможности

### Основные
- **Задачи протокола** — тема, номер тикета, приоритет, состояние, срок, исполнители.
- **История статусов** — у каждой задачи неограниченное число статусов (дата + описание).
  В таблице показываются 3 последних в аккордеоне, полная история — в отдельном окне.
- **Состояния**: в работе / отложено / закрыто. Переключаются кликом на тайле дашборда.
- **Приоритеты**: высокий / средний / низкий (цветовой маркер).
- **Справочники** отделов и исполнителей; исполнитель привязывается к отделу.

### Фильтрация и поиск
- **Дашборд с фильтрами** — четыре кликабельных тайла (Все / В работе / Отложено / Закрыто)
  отфильтровывают таблицу по состоянию.
- **Расширенные фильтры** — по отделу, исполнителю и приоритету одновременно.
- **Поиск** по теме, тикету, исполнителю и отделу (поле в правом углу шапки).
- **Сортировка** по столбцам (кликом на заголовок).
- **Запоминание вида** — фильтры, поиск, сортировка сохраняются в `localStorage` и восстанавливаются при перезагрузке.

### Интерактивность
- **Быстрое добавление статуса** прямо из строки таблицы (кнопка `+`).
- **Экспорт / импорт** в CSV (совместимо с Excel).
- **Печать** — кнопка в тулбаре вызывает `Ctrl+P` с аккуратным @media print стилем.

### Справочники
- **Группировка исполнителей** по отделам в списке справочника.
- **Счётчики** — число задач per исполнитель, число исполнителей per отдел.
- **Поиск** в списке исполнителей (по имени и отделу).
- **Инлайн-редактирование** отделов и исполнителей.

## Технологический стек

| Слой        | Технология                                              |
|-------------|---------------------------------------------------------|
| Backend     | Python 3.12, [FastAPI](https://fastapi.tiangolo.com/)   |
| ASGI-сервер | Uvicorn                                                 |
| Валидация   | Pydantic v2                                             |

## Быстрый старт

Требуется установленный Docker.

### Запуск контейнера

```bash
docker compose up -d --build
```

Приложение доступно на **http://localhost:8000**.

### Проверка (smoke-тест)

```bash
bash .claude/skills/run-protocol/driver.sh
```

Этот скрипт проверяет, что:
- Web UI загружается
- API endpoints отвечают
- База данных доступна

### Полезные команды

```bash
docker compose logs -f        # логи в реальном времени
docker compose stop           # остановить
docker compose start          # запустить снова
docker compose down           # остановить и удалить контейнер (данные сохраняются)
```

**Данные** хранятся в `./data/protocol.db` (примонтированы через volume),
поэтому пересборка и перезапуск контейнера их не теряют.

### Claude Code Integration

Проект поставляется с **skill** для автоматизированного запуска через Claude Code:

```bash
/run-protocol
```

Skill находится в `.claude/skills/run-protocol/` и содержит:
- **SKILL.md** — полная документация для агентов
- **driver.sh** — smoke-тест, проверяющий функциональность

Агент сможет собрать приложение, запустить его и проверить работоспособность
без дополнительных инструкций.

## Модель данных

Четыре таблицы SQLite. Схема создаётся и мигрируется автоматически при старте
(`init_db()` в [app.py](app.py)).

### `items` — задачи протокола

| Поле             | Тип     | Описание                                              |
|------------------|---------|-------------------------------------------------------|
| `id`             | INTEGER | PK                                                    |
| `topic`          | TEXT    | Тема задачи (обязательно)                             |
| `ticket`         | TEXT    | Номер тикета (необязательно)                          |
| `priority`       | TEXT    | `high` / `medium` / `low` / NULL                      |
| `state`          | TEXT    | `in_progress` / `postponed` / `closed`                |
| `due_date`       | TEXT    | Срок (ISO-дата `YYYY-MM-DD`)                          |
| `executors_json` | TEXT    | JSON-массив **ID** исполнителей, напр. `[3, 7]`       |
| `created_at`     | TEXT    | Дата создания                                         |

> Исполнители хранятся как массив ID, ссылающихся на `executors.id`.
> При отдаче через API ID разворачиваются в объекты `{id, name, department_name}`
> функцией `resolve_executors()`.

### `statuses` — история статусов

| Поле          | Тип     | Описание                                          |
|---------------|---------|---------------------------------------------------|
| `id`          | INTEGER | PK                                                |
| `item_id`     | INTEGER | FK → `items.id`, `ON DELETE CASCADE`              |
| `status_date` | TEXT    | Дата статуса                                      |
| `status_note` | TEXT    | Текст статуса                                      |
| `created_at`  | TEXT    | Время добавления                                  |

### `departments` — отделы

| Поле   | Тип     | Описание                |
|--------|---------|-------------------------|
| `id`   | INTEGER | PK                      |
| `name` | TEXT    | Название (UNIQUE)       |

### `executors` — исполнители

| Поле            | Тип     | Описание                                            |
|-----------------|---------|-----------------------------------------------------|
| `id`            | INTEGER | PK                                                  |
| `name`          | TEXT    | ФИО / название (UNIQUE)                             |
| `department_id` | INTEGER | FK → `departments.id`, `ON DELETE SET NULL`         |

### Связи

```
departments 1───* executors
                      ▲
                      │ (по ID в executors_json)
items 1───* statuses  │
   └──────────────────┘  (многие-ко-многим через JSON-массив)
```

## REST API

Базовый префикс: `/api`. Все тела запросов/ответов — JSON (кроме импорта CSV).

### Задачи

| Метод    | Путь                | Описание                                          |
|----------|---------------------|---------------------------------------------------|
| `GET`    | `/api/items`        | Список задач. Параметр `?state=` фильтрует.        |
| `POST`   | `/api/items`        | Создать задачу (опц. сразу первый статус).         |
| `PUT`    | `/api/items/{id}`   | Частичное обновление (только переданные поля).      |
| `DELETE` | `/api/items/{id}`   | Удалить задачу вместе со статусами.                |

Ответ `GET /api/items` для каждой задачи дополнительно содержит:
- `executors` — массив объектов `{id, name, department_name}`;
- `recent_statuses` — до 3 последних статусов (от новых к старым).

**Особенности `PUT`:**
- передаются только изменяемые поля (`exclude_unset`);
- пустое значение **очищает** поле в БД — кроме `topic` и `state` (защищены от обнуления);
- `executors` — массив ID; пустой массив очищает исполнителей.

### Статусы

| Метод    | Путь                          | Описание                          |
|----------|-------------------------------|-----------------------------------|
| `GET`    | `/api/items/{id}/statuses`    | Полная история статусов задачи.   |
| `POST`   | `/api/items/{id}/statuses`    | Добавить статус.                  |
| `DELETE` | `/api/statuses/{id}`          | Удалить отдельный статус.         |

### Отделы

| Метод    | Путь                       | Описание                                                  |
|----------|----------------------------|-----------------------------------------------------------|
| `GET`    | `/api/departments`         | Список отделов.                                           |
| `POST`   | `/api/departments`         | Создать отдел (409, если имя занято).                     |
| `DELETE` | `/api/departments/{id}`    | Удалить; у исполнителей отдела `department_id` → NULL.    |

### Исполнители

| Метод    | Путь                     | Описание                                       |
|----------|--------------------------|------------------------------------------------|
| `GET`    | `/api/executors`         | Список с именем отдела (сортировка отдел→имя).  |
| `POST`   | `/api/executors`         | Создать (`{name, department_id?}`, 409 при дубле). |
| `DELETE` | `/api/executors/{id}`    | Удалить из справочника (задачи не затрагиваются). |

### Экспорт / импорт

| Метод  | Путь               | Описание                                  |
|--------|--------------------|-------------------------------------------|
| `GET`  | `/api/export/csv`  | Скачать все задачи в CSV (UTF-8 BOM).      |
| `POST` | `/api/import/csv`  | Загрузить CSV (`multipart/form-data`).     |

> Интерактивная документация API доступна по адресу **http://localhost:8000/docs**
> (Swagger UI, генерируется FastAPI автоматически).

## Устройство фронтенда

Весь интерфейс — в [static/index.html](static/index.html). Структура:

- **`<style>`** — все стили инлайн. Цветовая схема построена вокруг тёмной шапки
  (`#1a1a2e`) и синего акцента (`#2563eb`).
- **Две вкладки** (`tab-protocol`, `tab-refs`), переключаются JS без перезагрузки.
- **`<script>`** — нативный JS без зависимостей. Ключевые блоки помечены
  комментариями-разделителями (`// ── … ──`).

Состояние приложения в глобальных переменных:

| Переменная       | Назначение                                |
|------------------|-------------------------------------------|
| `allItems`       | Все задачи (кэш с сервера)                 |
| `allExecutors`   | Справочник исполнителей                    |
| `allDepartments` | Справочник отделов                         |
| `currentFilter`  | Активный фильтр по состоянию               |
| `searchQuery`    | Текущая строка поиска                      |
| `sortState`      | Колонка и направление сортировки           |

Основной поток: `loadAll()` тянет данные с трёх эндпоинтов параллельно
(`Promise.all`), затем `render()` перерисовывает таблицу с учётом
фильтра / поиска / сортировки (`applyFilters` → `applySort`).

Поля исполнителей в форме задачи строятся динамически (`buildExecutorSelect`):
`<select>` с группировкой по отделам через `<optgroup>`.

## Миграции БД

Все миграции выполняются в `init_db()` при старте приложения и **идемпотентны**
(повторный запуск безопасен). Подход: проверка наличия колонки через
`PRAGMA table_info`, затем `ALTER TABLE` или пересоздание таблицы.

Исторические миграции (см. [CHANGELOG.md](CHANGELOG.md)):
1. Перенос статусов из старой одностолбцовой схемы `items` → таблица `statuses`.
2. Добавление колонок `ticket`, `priority`, `executors_json`.
3. Перенос `executor` / `executor2` → `executors_json`.
4. Перевод исполнителей в `executors_json` из имён (строк) в **ID**.

> При добавлении новой колонки или таблицы — дописывайте миграцию в `init_db()`,
> не полагаясь на пересоздание БД (у пользователя уже есть данные).

## Экспорт / импорт CSV

Формат столбцов (`CSV_FIELDS` в [app.py](app.py)):

```
id, topic, ticket, priority, state, due_date, executors, last_status_date, last_status_note
```

- **Кодировка**: UTF-8 с BOM — чтобы Excel корректно открывал кириллицу.
- **Исполнители**: в одной ячейке через ` | `, в формате `Отдел — Имя`.
- **Импорт**: `id` игнорируется (создаются новые записи); исполнители
  сопоставляются по имени, ненайденные создаются без отдела; экспортируется
  и импортируется только последний статус задачи.

## Разработка

### Запуск без Docker

```bash
pip install -r requirements.txt
uvicorn app:app --reload --port 8000
```

БД по умолчанию создаётся в `data/protocol.db`. Путь переопределяется
переменной окружения `DB_PATH`.

### Изменение фронтенда

Правьте `static/index.html` напрямую. При запуске через Docker файл копируется
в образ — после изменений нужна пересборка:

```bash
docker compose up -d --build
```

Для частой работы над фронтендом удобнее запускать бэкенд локально с `--reload`
(uvicorn отдаёт `static/` напрямую с диска — пересборка не нужна).

### Конвенции

- Бэкенд: каждая функция-эндпоинт открывает соединение через `get_db()`
  и закрывает его перед возвратом. Комментарии — на русском, поясняют назначение.
- Фронтенд: без сборки и npm-зависимостей. Новый код — в стиле существующих
  блоков, с комментариями-разделителями.
- Любое заметное изменение фиксируется в [CHANGELOG.md](CHANGELOG.md)
  (формат [Keep a Changelog](https://keepachangelog.com/ru/1.1.0/)).

## Резервное копирование

Вся база — один файл `data/protocol.db`. Достаточно его скопировать:

```bash
cp data/protocol.db backup-$(date +%Y%m%d).db
```

Восстановление — заменой файла (при остановленном контейнере).
Альтернатива — выгрузка через `GET /api/export/csv`.
